What makes this safe without human confirmation
Degraded fallback chain
Every failure has an automatic recovery. Can’t detect dates? Regex. Orphaned rows? Flatten. AI timeout? Fall back to deterministic parsing. System always commits something — problems produce lower completeness, not errors.
Learning loop
Every correction feeds back via ai_suggestion_logs. After 50+ corrections per builder, few-shot examples injected per-customer. Header hash cache means repeat uploads skip AI entirely — 100% accuracy on known formats.
Anomaly detection
Lives in lookahead_engine.py. On recalculation after new programme version: compares snapshots. Demand doubles or 40%+ classifications shift = anomaly_flags written to lookahead_snapshots. PM dashboard reads via /api/lookahead. Notifications table stays sub-only.
⚠ Conscious decision: auto-commit in construction context
This system has no human sign-off step. Every auto-committed mapping is tagged with confidence tier, source, and timestamp. The audit trail exists — it just doesn’t include manual confirmation. For MVP on one known project (Bowden, P6 format), the fallback chain + anomaly detection provides sufficient safety. At scale with unknown files, medium-confidence auto-commits will produce silent errors. If this causes problems post-MVP, adding an optional confirmation toggle per-project is a 1-day change, not an architecture change.
